# 构建时序报告（Build Timeline Report）

目的：让“从初始化到上线与后续迭代”的构建/发布节奏、关键变更与风险可追踪，避免后期口口相传导致返工。

约定：
- 每次**可交付构建**（TestFlight/内测包/灰度/正式）都新增一条记录。
- 每条记录都要包含：构建编号、时间、提交范围、破坏性变更、回滚策略、验证清单。
- 这份文档长期跟随项目周期维护；变更由提交 PR 的人同步更新。

---

## 0. 里程碑（Milestones）

- M0：项目骨架完成（主题/tokens/路由/CI）
- M1：首个可用流程（登录/首页/关键功能之一）
- M2：内测（Crash/性能/埋点闭环）
- M3：首发（合规/隐私/商店素材齐备）
- M4：迭代（版本化与回滚机制成熟）

---

---

## 2. 构建时序（Build Sequence）

- 开发阶段：本地调试 → 单元测试/静态分析 → 设备回归
- 集成阶段：合并到主线 → 自动化构建 → 产物归档
- 发布阶段：内测 → 灰度 → 正式发布
- 运营阶段：监控（Crash/性能）→ 回滚/热修 → 版本迭代

---

## 3. 规则与审计（Governance）

- 版本号策略：SemVer（建议）或按产品约定
- Build number 规则：平台自增 + 可追溯到 Git
- 变更门禁：格式化、静态分析、（未来）测试必须通过
- 责任人：每次发布由发布负责人确认本报告已更新
